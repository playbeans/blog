设计模式分类

  1. 创建模式

工厂模式,抽象工厂,生成器模式,原型模式

  1. 结构模式

适配器模式,桥接模式,组成模式,装饰模式,外观模式,享元模式,代理模式

  1. 行为模式

中介模式,命令模式,备忘录模式,观察者模式,状态模式,策略模式,迭代器模式,访问者模式,责任链模式

6大设计原则

  1. 单一职责:一个类只完成一个功能,如果做不到类只完成一个功能,最少要保证一个方法完成一个功能.Single Responsibility Principle，简称是SRP。
  2. 依赖倒置:高层依赖应该依赖接口,而不应该依赖具体的实现.Dependence Inversion Principle,DIP
  3. 里氏替换:凡是父类出现的地方都可以用子类替换,并且原功能没有发生变化,子类不应该覆盖父类的抽象方法.Liskov Substitution Principle，LSP
  4. 迪米特法则:一个类要尽量的封装自己,一个类只与自己的朋友类打交道.Law of Demeter，LoD
  5. 接口隔离原则:一个接口完成后功能尽可能单一,不要让一个接口承担过多的责任.Interface Segregation Principle
  6. 开闭原则:对扩展开放,对修改关闭Open Closed Principle

抽象工厂

建造者模式

把一个复杂的对象的构建与表示分开,使得构建方法相同,构建顺序不同,构建出来的类也不同

代理设计模式

代理模式分为两种,普通代理,和强制代理,

普通代理:可以直接通过代理类去代理类

强制代理:一定要找到该类,才能使用该类的代理方法

Spring IOC 和APO都使用了代理设计模式

原型设计模式

通过对象复制来实现的模式就是原型模式,原型模式的核心试试clone方法

优点:性能优良,比new一个对象性能要好.

中介模式

封装一系列对象的交互,由原先的网格状交互变成星状交互.

优点:减少类与类之间的依赖,

缺点:中介者类会变得很膨胀

命令模式

类之间的解耦,容易扩展,一般GUI开发,一个按钮就是一个点击命令

Receiver:接收者，干活的角色，命令执行到这里被执行

Command：所有执行者在这里声明

Invoker：调用者执行Command的抽象方法

责任链模式

责任连的优点是把请求和处理分开,请求者可以不知道谁处理,处理者可以不用知道请求的全貌.

缺点是:性能问题,每个请求从头到尾,调试不方便,一般做法是设置一个最长调用链

装饰模式

动态的给一个对象添加额外的职责,

使用场景:

  1. 扩展一个了类,给一个类增加附加功能
  2. 动态给一个类增加功能,这些功能可以再动态的撤销

缺点:

  1. 多层的装饰比较复杂,不容易排查问题

ConcreteStrategy：具体策略角色

适配器模式

一个不同的类，通过继承修改同时满足另一个类的要求。

适配器是一补救模式，通常解决接口不相容的问题，

观察者模式

观察者模式也叫作发布订阅模式,它是一个在项目中经常使用到的模式.(定义对象间一种一对多的依赖关系,使得每当一个对象改变状态,则依赖的对象都会得到通知并且更新)

RabbitMQ简介
  1. Mesage:消息,由消息头和消息体组成
  2. Publisher:笑死生产者
  3. Exchange(将消息路由给队列):交换机
  4. Binding:消息队列和交换机之间的关联
  5. Queue:消息队列,一个消息可以投入一个或者多个队列
  6. Connection:比如一个TCP链接
  7. Channel:多路复用连接中一条独立的双向数据流通道.因为Connection链接昂贵,所以公用一条
  8. Consumer:消息的消费者,从消息队列中取得消息的客户端应用
  9. Virtual Host:一批交换机,消息队列,关联对象的配置
  10. Broker:消息队列服务实体
RabbitMQ架构图
Exchange类型
  1. Direct:直接发送,单播模式,echange和binding的routingkey完全相同
  2. Fanout:广播分发,把消息发送到每个队列上
  3. Topic:一个消息匹配对应的Queue队列进行发送消息，带有通配符
  4. Header:匹配消息头，根据消息header进行判断具体转发那个queue

网络

7层架构
  1. 物理层:定义物理设备标准,传输bt流,把1,0转化成电流交互
  2. 数据链路层:从物理层接受的数据进行mac地址封装与解封,这一层的设备叫交换机
  3. 网络层:下层接受的地址进行IP地址的解封和封装,这一次的设备是路由器
  4. 传输层:定义传输数据的协议和端口,如TCP,UDP,
  5. 会话层:通过传输层建立数据传出通道.
  6. 表示层:对接收的数据进行加密,解密,压缩,解压缩,把计算机识别的东西转化成人能够识别的东西
  7. 应用层:终端设备WEB,FTP,WEB,QQ
三次握手
  1. 客户端发送带有SYN标示的请求
  2. 服务端回应客户端SYN+ACK的回应
  3. 客户端发送ACK再次确认.
四次挥手
  1. 客户端发送FIN=M关闭服务端请求,
  2. 服务端发送ACK=M+1相应客户端
  3. 服务端发送FIN=N请求断开链接
  4. 客户端回应ACK=N+1.

Get和Post区别
  1. Get是不安全的，因为在传输过程，数据被放在请求的URL中；Post的所有操作对用户来说都是不可见的。
  2. Get传送的数据量较小，这主要是因为受URL长度限制；Post传送的数据量较大，一般被默认为不受限制。
  3. Get限制Form表单的数据集的值必须为ASCII字符；而Post支持整个ISO10646字符集。
  4. Get执行效率却比Post方法好。Get是form提交的默认方法。
  5. GET产生一个TCP数据包；POST产生两个TCP数据包。（非必然，客户端可灵活决定）
Http请求的完全过程
  1. 浏览器根据域名解析IP地址（DNS）,并查DNS缓存
  2. 浏览器与WEB服务器建立一个TCP连接
  3. 浏览器给WEB服务器发送一个HTTP请求（GET/POST）：一个HTTP请求报文由请求行（request line）、请求头部（headers）、空行（blank line）和请求数据（request body）4个部分组成。
  4. 服务端响应HTTP响应报文，报文由状态行（status line）、相应头部（headers）、空行（blank line）和响应数据（response body）4个部分组成。
  5. 浏览器解析渲染
tcp和udp区别
  1. TCP面向连接，UDP是无连接的，即发送数据之前不需要建立连接。
  2. TCP提供可靠的服务。也就是说，通过TCP连接传送的数据，无差错，不丢失，不重复，且按序到达;UDP尽最大努力交付，即不保证可靠交付。
  3. TCP面向字节流，实际上是TCP把数据看成一连串无结构的字节流，UDP是面向报文的，UDP没有拥塞控制，因此网络出现拥塞不会使源主机的发送速率降低（对实时应用很有用，如IP电话，实时视频会议等）
  4. 每一条TCP连接只能是点到点的，UDP支持一对一，一对多，多对一和多对多的交互通信。
  5. TCP首部开销20字节，UDP的首部开销小，只有8个字节。
  6. TCP的逻辑通信信道是全双工的可靠信道，UDP则是不可靠信道。
tcp和udp的优点
  • TCP的优点： 可靠，稳定 TCP的可靠体现在TCP在传递数据之前，会有三次握手来建立连接，而且在数据传递时，有确认、窗口、重传、拥塞控制机制，在数据传完后，还会断开连接用来节约系统资源。 TCP的缺点： 慢，效率低，占用系统资源高，易被攻击 TCP在传递数据之前，要先建连接，这会消耗时间，而且在数据传递时，确认机制、重传机制、拥塞控制机制等都会消耗大量的时间，而且要在每台设备上维护所有的传输连接，事实上，每个连接都会占用系统的CPU、内存等硬件资源。 而且，因为TCP有确认机制、三次握手机制，这些也导致TCP容易被人利用，实现DOS、DDOS、CC等攻击。
  • UDP的优点： 快，比TCP稍安全 UDP没有TCP的握手、确认、窗口、重传、拥塞控制等机制，UDP是一个无状态的传输协议，所以它在传递数据时非常快。没有TCP的这些机制，UDP较TCP被攻击者利用的漏洞就要少一些。但UDP也是无法避免攻击的，比如：UDP Flood攻击…… UDP的缺点： 不可靠，不稳定 因为UDP没有TCP那些可靠的机制，在数据传递时，如果网络质量不好，就会很容易丢包。 基于上面的优缺点，那么： 什么时候应该使用TCP： 当对网络通讯质量有要求的时候，比如：整个数据要准确无误的传递给对方，这往往用于一些要求可靠的应用，比如HTTP、HTTPS、FTP等传输文件的协议，POP、SMTP等邮件传输的协议。 在日常生活中，常见使用TCP协议的应用如下： 浏览器，用的HTTP FlashFXP，用的FTP Outlook，用的POP、SMTP Putty，用的Telnet、SSH QQ文件传输。什么时候应该使用UDP： 当对网络通讯质量要求不高的时候，要求网络通讯速度能尽量的快，这时就可以使用UDP。 比如，日常生活中，常见使用UDP协议的应用如下： QQ语音 QQ视频 TFTP。

  3. ConfigurableBeanFactory:一个重要的借口,增强IOC容器的可定制性,设置类装载器,属性编辑器,容器初始化后置处理等方法.
  4. ListableBeanFactory:访问Bean基本信息的若干方法,如查看Bean的个数,获取某一类型的Bean.

  6. SingletonBeanRegistry:运行期间注册单例Bean,第二次使用getBean直接从缓存中取,
Spring作用域

Spring定义了5种作用域,singleton(单例),prototype(原型),request,sesstion,global session.

  1. singleton:单例模式(多线程不安全),Spring缺省作用域
  2. protype(原型):每次通过Spring容器获取protype定义的bean时,容器都将创建一个新的bean实例,每个bean都有自己的属性和状态.根据经验对有状态的bean使用protype,对无状态的bean使用singleton.
  3. request(一次请求一个实例):当前的bean仅仅在当前的http请求中有效.当前Http请求结束,该bean实力会被销毁.
  4. sesstion
  5. global Seesion:在一个全局的Http Session中,容器会返回该Bean的同一个实力
SpringBean生命周期
  1. 实例化一个Bean,也就是New
  2. Spring上下文实例化,IOC容器注入.
  3. Bean实现beanNameAware,调用setBeanName方法,此处传递spring配置文件中Bean的Id
  4. BeanFactoryAware:如果实现BeanFactoryAware,就会调用它实现的setBeanFacoty,传递的是Spring工厂本身.
  5. ApplicationContextAware:如果这个类实现了ApplicationContextAware,会调用setApplicationContext方法,传入Spring上下文.
  6. postProcessBeforInitialization:接口实现初始化预处理,如果这个Bean关联了BeanPostProcessor接口,将会调用postProcessBeforeInitialization(Object,String)方法,BeanPostProcess经常用作Bean内容的更改,并且由于这个是在Bean初始化结束调用的方法,也可以应用于内存或缓存技术.
  7. init-method:执行在配置中配置的init-method方法
  8. postProcessAfterInitialization:如果bean关联了postprocess接口,执行postProcessAfterInitialization,工作完成之后就可以使用这个Bean了
  9. Destory:清理不用的bean
  10. destory-method:配置清理,如果配置了就调用.

事物引发的安全问题

脏读:读到另外一个事物没有提交的事物

不可重复读:读到另外一个事物更新的数据,多次查询导致数据不一致

幻读:一个事物读取到另外一个事物已经insert的数据,多次查询导致不一致

事物的隔离级别
  1. 未提交读（Read UnCommited）:脏读,不可重复读,幻读都有可能发生
  2. 已提交读:避免脏读,不可重复读,幻读会发生.
  3. 可重复读:脏读,不可重复读不会发生.幻读肯能发生
  4. 串行化:可以避免所有问题
事物的传播
  1. Required:支持事务,不存在就新建一个.
  2. Required_New:如果事物存在,就挂起线程,创建一个新的.
  3. Support:支持事物,如果不存在就不使用事务
  4. Not_Support:以非事物运行,如果存在就挂起
  5. Mandatory:一定有事物,否则就抛异常
  6. Never:以非事物方式运行,如果存在就跑出异常
  7. Nested:如果存在就以嵌套事物执行.

collection关系
List 介绍

​ List是有序的Collection,通常有三个实现ArrayList,Vector,LinkedList

  1. ArrayList,底层是数组实现,它允许对元素的快速随机访问,当从ArrayList中间位置插入和删除元素时,需要堆数据进行复制和移动,代价比较大.因此它适合随机查找和遍历,不适合插入和删除.
  2. Vector:数组+线程同步,因为有同步锁,所以访问速度不如ArrayList
  3. ListList:使用链表做存储结构,适合数据的动态插入和删除,随机访问和遍历速度比较慢.另外可以操作表头和表尾,可以当做堆栈,队列,双向队列使用.
Set 介绍

内容无序，如果让两个对象相等一定要覆盖Objects的equals和hashCode,

  1. HashSet:无序,不可重复,底层是hashMap,hashCode相等值未必相等.
  2. TreeSet:有序,不可重复,底层是SortMap,使用的对象一定实现Comparabel接口.
  3. LinkHashSet:保证插入的顺序,不可重复.底层是hashSet和linkHashMap
