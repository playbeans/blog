6大设计原则

  1. 单一职责:一个类只完成一个功能,如果做不到类只完成一个功能,最少要保证一个方法完成一个功能.Single Responsibility Principle，简称是SRP。
  2. 依赖倒置:高层依赖应该依赖接口,而不应该依赖具体的实现.Dependence Inversion Principle,DIP
  3. 里氏替换:凡是父类出现的地方都可以用子类替换,并且原功能没有发生变化,子类不应该覆盖父类的抽象方法.Liskov Substitution Principle，LSP
  4. 迪米特法则:一个类要尽量的封装自己,一个类只与自己的朋友类打交道.Law of Demeter，LoD
  5. 接口隔离原则:一个接口完成后功能尽可能单一,不要让一个接口承担过多的责任.Interface Segregation Principle
  6. 开闭原则:对扩展开放,对修改关闭Open Closed Principle

23种设计模式

工厂模式:
抽象工厂
模板方法
建造者模式

把一个复杂的对象的构建与表示分开,使得构建方法相同,构建顺序不同,构建出来的类也不同

代理设计模式

代理模式分为两种,普通代理,和强制代理,

普通代理:可以直接通过代理类去代理类

强制代理:一定要找到该类,才能使用该类的代理方法

Spring IOC 和APO都使用了代理设计模式

原型设计模式

通过对象复制来实现的模式就是原型模式,原型模式的核心试试clone方法

优点:性能优良,比new一个对象性能要好.

中介模式

封装一系列对象的交互,由原先的网格状交互变成星状交互.

优点:减少类与类之间的依赖,

缺点:中介者类会变得很膨胀

命令模式

类之间的解耦,容易扩展,一般GUI开发,一个按钮就是一个点击命令

Receiver:接收者，干活的角色，命令执行到这里被执行

Command：所有执行者在这里声明

Invoker：调用者执行Command的抽象方法

责任链模式

责任连的优点是把请求和处理分开,请求者可以不知道谁处理,处理者可以不用知道请求的全貌.

缺点是:性能问题,每个请求从头到尾,调试不方便,一般做法是设置一个最长调用链

装饰模式

动态的给一个对象添加额外的职责,

使用场景:

  1. 扩展一个了类,给一个类增加附加功能
  2. 动态给一个类增加功能,这些功能可以再动态的撤销

缺点:

  1. 多层的装饰比较复杂,不容易排查问题
策略模式

Context封装角色：

Strategy：抽象策略角色

ConcreteStrategy：具体策略角色

适配器模式

一个不同的类，通过继承修改同时满足另一个类的要求。

适配器是一补救模式，通常解决接口不相容的问题，

迭代器设计模式

循环输出

组合模式

组合模式也叫合成模式,有时也叫部分整体模式,主要用来藐视部分和整体的关系,只要是体现局部跟整体的关系的时候,都可以考虑下组合模式.

观察者模式

观察者模式也叫作发布订阅模式,它是一个在项目中经常使用到的模式.(定义对象间一种一对多的依赖关系,使得每当一个对象改变状态,则依赖的对象都会得到通知并且更新)

  1. Subject:被观察者
  2. Observer:观察者
  3. ConcreteSubject:具体被观察者
  4. ConcreteObserver:具体的观察者

优点:建立一套触发机智,

缺点:触发机制连过长,开发调试比较难,一个观察者卡壳,影响整体的执行效率,通常使用异步的方式.

实践:

  1. 文件系统,
  2. AMT取款机
门面模式

门面模式也叫做外观模式,是一种比较常用的封装模式

优点:

减少系统见的相互依赖,
提高灵活性
提高安全性

缺点:

不符合开闭原则,对修改开放,对扩展关闭,稍有改动,影响极大

场景:

  1. 为一个复杂的模块或子系统提供一个外界访问的借口
  2. 子系统相对外界比较独立
  3. 预防低水平开发人员带来的风险扩散
备忘录模式

使用场景:

  1. 可以保存恢复数据的操作,
  2. 提供回滚的操作
  3. 数据库的事物管理就是备忘录模式

注意事项:

  1. 不要频繁的创建备忘对象,因为建立大对象需要消耗资源
访问者模式
  1. Visitor:抽象者
  2. ConcreteVisitor:具体抽象者
  3. Element:抽象元素
  4. ConcreteElement:具体元素
  5. ObjectStruture:结构对象

优点:

  1. 符合单一原则
  2. 优秀的扩展性
  3. 灵活度高

缺点:

  1. 访问类关注了其他类的内部,不符合迪米特法则
  2. 具体元素变更比较困难
  3. 报表运算,UI展示

场景:

对迭代器模式的扩充,可以遍历不同的类

状态模式
  1. State:抽象状态
  2. ConcreteState:具体状态
  3. Context:环境角色

状态模式引起的行为变更,看起来就好像这个对象对应的类发生一样.

优点:

  1. 避免多过得的if else语句

  2. 封装性好

  3. 遵循设计原则

    场景:

    1. 行为随着状态的改变而改变
    2. 大量的if else语句替换着
解释器模式

解释器是一个简单的语法分析工具,他的有点事扩展性,

使用最少,引起效率问题.

享元模式

减少内存的分配

场景:

  1. 大量相似的对象
  2. 需要缓冲池的场景
  3. d

策略模式和命令模式的区别
